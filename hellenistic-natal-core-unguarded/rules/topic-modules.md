Purpose: define the topic-module composite-reading architecture (Stage 2
of the v2.2 restructured workflow). Each **topic module** is a per-topic
composite reading that re-views the chart through a topic-specific lens
(太极点 model). This file is the **framework / catalog** layer:
per-topic specifications are filled by B.2, output format by B.4,
Stage 3 synthesis by B.5, orchestration by B.7.

## Concept (per v2.2 Plan B §0.1)

A **topic module** is a per-topic composite reading that:

1. Selects **relevant chart factors** (太极点 集合) for that topic.
2. Applies **topic-specific lens reframing** (same planet, different
   topic → different interpretation register).
3. Reads composite as **one integrated voice** about that topic.
4. Outputs concrete category brief shots (per Plan C output discipline,
   pending).
5. One module = one Stage 2 session = one output file.

Topic modules **重复访问 chart 但每次透镜不同** — the same planet may
appear in 4–5 topic modules with different register each time. The
Stage 1 deterministic data table (per Plan A) is the **single source**
queried by all modules.

Topic modules are **layer-above-focus-row**: they **reuse**
`significator-hierarchy.md` focus rows as building blocks but **do
not replace** them. The legacy single-session workflow
(`reading-workflow.md` step 0-7) remains valid for single-focus reading;
topic modules are an **additive** architecture for composite multi-topic
reading.

Lens reframing is **not placement cookbook** ("Mars in Leo = power" is
exactly the anti-pattern this architecture must avoid). Modules still
route through the audit-first gate per `planet-condition-audit.md`;
lens shifts the interpretation **register**, not the evidence chain.

### 4.1.4 Slug convention

- **ASCII lowercase letters + hyphens**. No underscores, spaces, or
  CJK characters in slugs.
- **1–3 词 maximum**, joined with hyphens. `love-marriage` ✓;
  `occult-spiritual-study-aptitude` ✗ — pick the shorter form
  (`occult`).
- **File naming pattern**: `cases/case-NN-topic-<slug>.md`. For
  example: `cases/case-07-topic-character.md`,
  `cases/case-07-topic-love-marriage.md`,
  `cases/case-07-topic-occult.md`.
- **Slugs coordinate with v1 focus-row vocabulary** where overlapping
  (see §4.1.3 coordination block).
- **Composite slugs make composite scope legible** when no 1:1
  mapping exists (e.g., `love-marriage` for 5+7; `family` for 4+10).

Slug rule provenance: `[project heuristic]`.

### 4.1.5 Module independence 原则

1. **Each module = its own Stage 2 session**. A session does not bundle
   multiple topic modules; each topic module is dispatched as a separate
   session output.
2. **Each session reads Stage 1 as input** (per Plan A, once Stage 1
   extraction lands). The Stage 1 data table is the **authoritative**
   source of chart positions, audits, reception network, configurations,
   Lots, cluster checks, and Same-Planet Multi-Role registry.
3. **Stage 2 sessions do not recompute Stage 1 data**. Re-deriving
   positions / audits / receptions inside a Stage 2 session is a
   violation; if Stage 1 is missing a required datum, **file an open
   thread**, do not silently recompute.
4. **Modules dispatch in parallel** (no inter-module dependency). All
   modules depend on Stage 1 being complete; once Stage 1 is in, the
   chosen Stage 2 modules can be dispatched as parallel sessions.
5. **Each session outputs to its own case file**:
   `cases/case-NN-topic-<slug>.md`. Modules do **not** write to each
   other's files.
6. **Stage 3 synthesis (per Plan B §8 / B.5) is sequential**, run
   after all chosen Stage 2 modules complete; it does **not** run in
   parallel with Stage 2.

Independence principle provenance: `[project heuristic]`.

### Composite reading framework

- **Integrate factors.** Read Asc + Asc ruler + sect light as the
  person-shape anchor triple. Layer each natural significator's audit
  (per `rules/planet-condition-audit.md`) as one psychic-faculty
  register. Apply Same-Planet Multi-Role handling (per
  `rules/significator-hierarchy.md` §"Same-Planet Multi-Role
  Handling") when Asc ruler coincides with sect light or with
  Lord-of-the-geniture.
- **Weight evidence.** Asc ruler condition + sect light condition +
  sect-light-to-Asc configuration are the top three weights. Angular
  planets carry visible-personality weight; cadent planets register
  as backdrop-tendency. Receptions to / from Asc ruler modulate but
  do not override raw audit.
- **Handle mixed-confidence.** Where benefic and malefic testimony
  converge on Asc ruler or sect light, output **mixed** with both
  named explicitly (do not average). Where placement and audit
  diverge (e.g., a planet strong by dignity but cadent + averse to
  sect light), report the divergence as a structural feature, not as
  a single net score.
- **Frame topic-specific output.** The character-module finding is
  about **who this person is as a whole person-shape**. Do not slip
  into career / love / wealth language. When a topic-specific finding
  emerges in passing (e.g., "this Mars suggests advisory work"),
  route it to the appropriate other module's open threads rather than
  including it here.

## Stage 2 Session Output Format (B.4 deliverable)

> ⚠ **v2.3 SUPERSEDED** (2026-05-17): see `rules/chart-finding-index-workflow.md` §1. This section preserved as v2.2 historical reference. v2.3 Stage 2 produces topic-level findings (per topic-finding 轻 schema), NOT 800-line narrative composite readings.

  - `NN` = chart number, two-digit padded to match the existing
    `cases/case-01-` through `cases/case-06a-` precedent.
  - `<slug>` = topic slug per §4.1.4 (ASCII lowercase, 1–3 词
    hyphen-joined). Use the registered slugs from §4.1.3 — `character`,
    `career`, `love-marriage`, `family`, `learning`, `wealth`, `occult`
    (the default 7), plus extension slugs `body`, `children`,
    `siblings`, `death`, `longevity`, `timing-overlay`. Slugs are

    time.

  rather than a 太极点 topic (per §"Topic: timing-overlay (slug:
  `timing-overlay`)" extension spec). When dispatched in mode (d), it

  timing-overlay dispatch protocol. All other extension modules in
  the catalog follow the standard `cases/case-NN-topic-<slug>.md`
  pattern.
- Examples: `cases/case-07-topic-character.md`,
  `cases/case-07-topic-career.md`,
  `cases/case-07-topic-love-marriage.md`.

  `cases/case-NN-topic-<slug>-TL.md` (full-Chinese, produced by the
  `translation-layer/` skill per `rules/brief-shot-discipline.md`
  §4.1.3.6 register split + this file's §"Framework" language
  convention). The technical Stage 2 file is the upstream artifact;
  the TL sibling derives from it. The two surfaces share the 8
  brief-shot criteria; only the surface language register differs.

````markdown

  evidence chain. The audit chain in
  `rules/planet-condition-audit.md` remains the gate (per

- Main narrative integrating the 太极点 factors and lens

  setup; trace-back is the durable handle the Stage 3 synthesis

  place) inside the composite reading is **not** a substitute for a

  as one structural feature with multiple role contributions

  opposing evidence.

  is the native, a person, an event, a state, or a life domain —

  per-alternative evidence cite + ≥1 explicit not-fit + hedging +

- Brief shots are **required** at the Stage 2 surface per
  `rules/brief-shot-discipline.md` §4.1.3.5 application context 1.

  layer**; brief-shot category names are written in **English** per

  reference catalog (合规 vs 不合规 examples; pattern summary at

- Four-tier framework — **High / Moderate / Low / Speculative** —
  plus **mixed**, per `rules/delineation-output-rules.md`
  §"Evidence Standard And Confidence Tiers".
- Mixed-evidence findings are reported as mixed, never averaged
  (per cross-module discipline preamble in this file; reinforced by
  `rules/delineation-output-rules.md` mixed-evidence handling).
- Confidence labels attach **per finding**, not per file. A single
  Stage 2 file can carry findings at different tiers.

## Open threads

- Topic-specific items requiring reader life-context to narrow.
- Items deferred to **other modules** (e.g., a career-module
  finding about partner-business angle deferred to `love-marriage`;
  a learning-module finding about ROI sub-question deferred to
  `wealth`) — name the receiving module and the deferred question.
- Items deferred to **specialist supplements** / extension modules
  (e.g., timing-conditional sub-questions routed through
  `timing-overlay` per §"Topic: timing-overlay").
- Items requiring further Stage 1 data (e.g., a
  degree-precision-dependent reading skipped because Stage 1 lacks
  the precision) — recorded as an open thread + a request to
  re-run the relevant Stage 1 step, per Discipline below
  (no-silent-Stage-1-fix bullet).
````

### Discipline

The discipline rules below govern every Stage 2 session file. Each
rule cites the upstream spec that defines it; this section does **not**
re-invent discipline, only assembles upstream rules into the Stage-2
application layer.

- **Cite Stage 1 row_ids, not chart factors by sign / degree.** Every
  finding in §"太极点", §"Lens reframing (applied)", and §"Composite
  reading" traces to one or more Stage 1 row_ids per
  `rules/stage1-data-extraction.md` §"Row identifier (row_id) schema"
  (`T1-<Planet>` / `T2-<From>-<To>-<type>-<mutual|oneway>` /
  `T3-<P1>-<P2>` / `T4-<Lot>` / `T5-<Sign>-<Place>` /
  `T6-<Planet>`). Citing a chart factor by sign / degree directly
  (e.g., "Venus at 18° Pisces") in place of a row_id is a
  recompute-style violation; the row_id is the durable handle the
  reading uses, not the underlying numerics.
- **Reception detection is single-source-of-truth at T2.** Reception
  relations are populated by the Stage 1 procedure in
  `rules/aspect-reception-rules.md` §"Reception Detection Procedure
  (v2.2 additive)" and recorded in Stage 1 Table 2. Stage 2 sessions
  read from T2; they do **not** re-detect receptions inside a
  session.
- **No recompute of Stage 1.** Stage 2 sessions do not re-derive
  positions, audits, receptions, aspects, Lots, cluster checks, or
  Multi-Role registrations — those are Stage 1 outputs (per §4.1.5
  module independence principle in this file). If a needed datum is
  missing or wrong, file an open thread + request a Stage 1 re-run
  (see no-silent-Stage-1-fix bullet below); do **not** silently
  recompute or silently patch.
- **Brief-shot discipline at the Stage 2 surface.** Brief shots
  satisfy the 8 criteria in `rules/brief-shot-discipline.md`
  §4.1.3.2 in full; brief shots are required at this surface per
  §4.1.3.5 application context 1; the language register for
  brief-shot category names at the Stage 2 surface is **English** per
  §4.1.3.6 (the TL sibling file owns the full-Chinese surface, per
  the language convention in this file's §"Framework" preamble).
- **Subject discipline.** The grammatical subject of every life
  claim in the file is a person / event / state / life domain —
  never a chart factor. This is the Anti-Cookbook rule from
  `rules/delineation-output-rules.md` §"Anti-Cookbook Rules" plus
  the "Chart factor as subject" anti-pattern in
  `rules/brief-shot-discipline.md` §4.1.3.3, applied at the Stage 2
  surface.
- **Per-topic ethical-guard inheritance.** Each Stage 2 session
  inherits the ethical-guard state declared in this file's
  §"Topic: `<slug>`" spec for the topic being run. The relevant
  declarations live at: §"Topic: family" → §"Ethical-guard
  reminder"; §"Topic: body" / §"Topic: children" / §"Topic:
  siblings" / §"Topic: death" / §"Topic: longevity" → §"Ethical-
  guard state"; §"Topic: timing-overlay" → ethical-guard
  inheritance note (the timing layer inherits the guard of each
  module it annotates). The Frame block restates the inherited
  state at the top of the file so the file's ethical posture is
  legible immediately.
- **No silent Stage 1 fix.** If a Stage 2 session detects a Stage 1
  gap, inconsistency, or error, the session **files an open
  thread** + **requests a Stage 1 re-run** through the handoff
  protocol — it does **not** silently patch the gap inside the
  Stage 2 file. This follows the Stage 1 → Stage 2 handoff protocol
  in `rules/stage1-data-extraction.md` §"Stage 1 → Stage 2 Handoff
  Protocol" plus the §A.2 Discipline bullet on row_id setup
  invariance.

## Stage 3 Synthesis Spec (B.5 deliverable)

> ⚠ **v2.3 SUPERSEDED** (2026-05-17): see `rules/chart-finding-index-workflow.md` §2. This section preserved as v2.2 historical reference. v2.3 Stage 3 produces `chart-finding-index.md` (per chart-level finding schema with embedded Stage 1 raw data + cross-link graph), NOT 1049-line portrait essay synthesis.


Stage 3 is the cross-topic synthesis layer: after all chosen Stage 2
topic-module sessions complete, one synthesis session integrates their
findings into a single whole-chart portrait. Stage 3 surfaces patterns
that emerge only when topic modules are read together (resonance +
tension across modules); it does **not** restate Stage 2 module content.
Stage 3's evidence chain runs through Stage 2 module references back to
Stage 1 row_ids — every Stage 3 cite must be mechanically traceable to a
Stage 1 row_id per `rules/stage1-data-extraction.md` §"Row identifier
(row_id) schema" (Gate 3 enforcement, defined below).

### Trigger

Stage 3 is dispatched **after all chosen Stage 2 topic sessions
complete**, per the user's menu choice in `skill-draft/SKILL.md`
§"First invocation flow" → §4 "Mode selection menu" (per B.6
deliverable). Concretely:

- **Mode (a) single-topic** (user picks one topic module): Stage 3 is
  **not** dispatched. A single-topic reading has no cross-topic surface
  to synthesize; the Stage 2 file is the terminal output.
- **Mode (b) multi-topic** (user picks a subset of the default 7
  modules): Stage 3 dispatched once all chosen Stage 2 sessions are
  complete and have passed their per-module output discipline.
- **Mode (c) full chart** (all default 7 modules + any extensions the
  user opted into): Stage 3 dispatched once every chosen Stage 2 session
  is complete.

Stage 3 is **never** dispatched concurrently with Stage 2 sessions still
in flight. Partial Stage 2 inputs produce a false-portrait risk (Stage 3
would synthesize from an incomplete topic surface and call it whole).
The orchestration owner is `SKILL.md` per B.6; this spec defines what
Stage 3 produces once dispatched, not the dispatch trigger mechanics.

### Discipline

The discipline rules below govern every Stage 3 portrait file. Each
rule cites the upstream spec that defines it; this section does **not**
re-invent discipline, only assembles upstream rules into the Stage-3
application layer.

- **Gate 3 enforcement (mandatory trace-back to Stage 1 row_id).**
  Every cite in §"整盘 Strengths", §"整盘 Difficulties", §"跨 topic
  Resonance", and §"跨 topic Tension" must trace back to **at least
  one Stage 1 row_id** per `rules/stage1-data-extraction.md` §"Row
  identifier (row_id) schema" (`T1-<Planet>` /
  `T2-<From>-<To>-<type>-<mutual|oneway>` / `T3-<P1>-<P2>` /
  `T4-<Lot>` / `T5-<Sign>-<Place>` / `T6-<Planet>`). Any cite that
  cannot be resolved to a corresponding Stage 1 row is a **Gate 3
  violation**: either (a) Stage 3 is asserting a fact that Stage 1 does
  not contain, or (b) Stage 1 is missing a row that should exist. Both
  cases → **STOP** the Stage 3 session, fix the source (Stage 3
  evidence chain or Stage 1 table) before resuming. Gate 3 is the
  downstream enforcement of the row_id schema set up by A.2 in
  `rules/stage1-data-extraction.md` §"Row identifier (row_id) schema"
  Gate 3 setup note (line: "Stage 3 mechanical trace-back enforcement
  is defined in v2.2 Plan B.5"). This is **not** a "please cite when
  convenient" rule; it is a mechanical STOP-and-fix protocol.
- **No repeat of Stage 2 content.** Stage 3 surfaces cross-topic
  patterns only; per-module composite-reading paragraphs are **not**
  restated in the portrait. If a portrait bullet would be readable as
  a verbatim restatement of one Stage 2 module's §"Composite reading"
  surface, it is a per-module finding misplaced at Stage 3 — drop it
  or refile to its source module's open threads.
- **No over-claim.** Cross-topic synthesis is an **interpretive
  overlay** on Stage 2 outputs, not a Stage 1-style fact. The
  portrait's strengths / difficulties / resonance / tension findings
  inherit the confidence-tier discipline of
  `rules/delineation-output-rules.md` §"Evidence Standard And
  Confidence Tiers" (High / Moderate / Low / Speculative + mixed);
  Low / Speculative cross-topic claims are **not** stated in High
  language.
- **Hedged language; conditional outcomes; reader-verification
  space.** Portrait prose uses conditional verbs ("tends to" /
  "likely" / "leans toward" / "倾向 / 可能") and reserves a
  reader-verification surface in §"Open threads for whole chart" for
  patterns whose resolution depends on life-context the chart cannot
  supply (per the §"Open threads" structural slot above + the §"Open
  Threads" honesty rule in `rules/delineation-output-rules.md`).
- **Mixed-evidence reported as mixed.** When ≥2 Stage 2 modules
  contribute opposing testimony on a single structural axis, the
  Stage 3 surface reports the result as **mixed** with both clusters'
  cites named; **never** averaged into a false middle (per
  `rules/delineation-output-rules.md` §"Evidence Standard And
  Confidence Tiers" well-evidenced-contradiction handling, applied at
  the cross-module surface).
- **Difficult conditions get pressure + mitigation + channel.** Every
  §"整盘 Difficulties" bullet — and any §"跨 topic Tension" bullet
  whose tension axis carries 6 / 8 / 12 / maltreatment / combust /
  contrary-to-sect malefic testimony from one or more contributing
  modules — reports **pressure domain + available mitigation +
  concrete constructive channel** per
  `rules/delineation-output-rules.md` §"Difficult-Condition
  Guardrails". Difficulty is **never** narrated as a doom verdict;
  difficulty of process is reported separately from value of outcome
  per same file §"Capacity Versus Outcome".
- **Subject discipline preserved.** The grammatical subject of every
  Stage 3 life claim is the native, a person, an event, a state, or
  a life domain — never a chart factor. This is the
  `rules/delineation-output-rules.md` §"Anti-Cookbook Rules" subject
  rule + the `rules/brief-shot-discipline.md` §4.1.3.3 §C.1
  anti-pattern "Chart factor as subject", applied at the Stage 3
  surface (and applied **inside** cite brackets too: even the
  bracketed module-reference example body uses person / state /
  domain subjects).
- **Per topic-module ethical-guard inheritance.** Stage 3 inherits the
  ethical-guard state of every Stage 2 module that fed it. If the
  chosen module set includes any extension module with an
  ethical-OFF default (`death` / `longevity` per §"Topic: death" /
  §"Topic: longevity" → §"Ethical-guard state"; `body` / `children` /
  `siblings` per their respective §"Ethical-guard state" blocks),
  Stage 3 **does not** re-enable that module's content into the
  synthesis surface unless explicit consent verification has been
  recorded in the Frame (per the consent-verification convention
  declared in the per-topic §"Ethical-guard state" blocks). An
  ethical-OFF module's findings stay outside the strengths /
  difficulties / resonance / tension surfaces; the Frame records the
  module was run-but-guarded, and the synthesis silently routes
  around it.
- **No silent Stage 2 fix.** If a Stage 3 session detects a problem
  in a contributing Stage 2 module output (e.g., a finding that
  cannot be Gate 3 traced back to Stage 1, an internal-discipline
  violation per §"Stage 2 Session Output Format" → §"Discipline",
  a mismatched ethical-guard state), the Stage 3 session **files an
  open thread** + **requests a Stage 2 re-run** through the
  orchestration handoff — it does **not** silently patch the gap
  inside the portrait file. This mirrors the no-silent-Stage-1-fix
  rule from `rules/stage1-data-extraction.md` §"Stage 1 → Stage 2
  Handoff Protocol" applied one stage upward.

## Orchestration Protocol (B.7 deliverable)

> ⚠ **v2.3 SUPERSEDED** (2026-05-17): see `rules/chart-finding-index-workflow.md` (and Phase 3 expansion for full orchestration). This section preserved as v2.2 historical reference. v2.3 orchestration uses chart-finding-index workflow with topic-findings parallel dispatch + finding-index consolidation; v2.2 4-mode menu structure may be revisited in Phase 3.

Stage dispatch sequence per mode. Modes correspond to
`skill-draft/SKILL.md` §"First invocation flow" → §4 "Mode selection
menu" choices. This section is the **dispatch playbook** between menu
selection and per-session handoff prompts; per-session prompt boilerplate
is in the sibling file `handoff-templates/v2.2-stage-handoffs.md`.

This section does **not** re-spec the Stage 1 → Stage 2 boundary (that
is `rules/stage1-data-extraction.md` §"Stage 1 → Stage 2 Handoff
Protocol", A.4 deliverable), the Stage 2 file structure (that is
§"Stage 2 Session Output Format", B.4 deliverable), the Stage 3
synthesis structure / trigger (that is §"Stage 3 Synthesis Spec", B.5
deliverable), or the parallel-dispatch independence principle (that is
§4.1.5 "Module independence 原则", B.1 deliverable). Orchestration cites
those specs and arranges them into a per-mode dispatch sequence.

### Mode (a) — Single topic

1. **Stage 1 session** → `chart-NN-stage1.md`.
2. **Stage 1 audit session** (independent fresh-eye session, not the
   Stage 1 author session) → `chart-NN-stage1-audit.md`. STOP whole
   flow if audit fail; do **not** dispatch Stage 2 until audit pass
   (per `rules/stage1-data-extraction.md` §"Stage 1 → Stage 2 Handoff
   Protocol").
3. **Stage 2 single-topic session** → `cases/case-NN-topic-<slug>.md`
   (per §"Stage 2 Session Output Format").
4. Stage 3 **not** dispatched (single topic has no cross-topic surface;
   per §"Stage 3 Synthesis Spec" → §"Trigger" mode (a) rule).

**Output**: 3 files (1 Stage 1 + 1 audit + 1 Stage 2). Optional TL
pass: +1 file (`cases/case-NN-topic-<slug>-TL.md`).

### Mode (b) — Multiple specific topics (≥2)

1. **Stage 1 session** → `chart-NN-stage1.md`.
2. **Stage 1 audit session** → `chart-NN-stage1-audit.md`; STOP if
   fail.
3. **Stage 2 sessions** (N parallel, N = chosen topic count): one per
   chosen topic slug → `cases/case-NN-topic-<slug>.md` per topic.
   Parallel dispatch per §4.1.5 "Module independence 原则".
4. **Stage 3 synthesis session** (sequential, after **all** N Stage 2
   sessions complete) → `cases/case-NN-portrait.md` (per §"Stage 3
   Synthesis Spec" → §"Trigger").

**Output**: 3+N files. Optional TL: +N (per Stage 2 file) and/or +1
(per portrait).

### Mode (c) — Full portrait

1. **Stage 1 session** → `chart-NN-stage1.md`.
2. **Stage 1 audit session** → `chart-NN-stage1-audit.md`; STOP if
   fail.
3. **Stage 2 sessions** (7 parallel — all default 7 topics per §4.1.3
   "Module catalog"): `character` / `career` / `love-marriage` /
   `family` / `learning` / `wealth` / `occult` → 7 files
   `cases/case-NN-topic-<slug>.md`.
4. **Stage 3 synthesis session** (sequential, after all 7 Stage 2
   sessions complete) → `cases/case-NN-portrait.md`.

**Output**: 10 files (1 Stage 1 + 1 audit + 7 Stage 2 + 1 Stage 3).
Optional TL: +up to 8 (7 per-topic + 1 portrait).

### Mode (d) — Custom

Same shape as mode (b), but the topic set may include extension modules
(`body` / `children` / `siblings` / `death` / `longevity` /
`timing-overlay`) per §4.1.3 "Module catalog".

**Sensitive module dispatch protocol** (`death` / `longevity`):

- **Before** dispatching the Stage 2 session for `death` or
  `longevity`, verify **explicit user consent** + the **symbolic /
  character-resonance frame** is declared (per `skill-draft/SKILL.md`
  §"First invocation flow" → §4 "Mode selection menu" sensitive-module
  handling + this file §"Topic: death" / §"Topic: longevity" →
  §"Ethical-guard state").
- If consent not verified → **skip** the module (mark as skipped in
  Stage 3's Frame "Topic modules run"); do **not** dispatch the Stage
  2 session.
- If dispatched, the Stage 2 session output carries the explicit
  ethical-guard banner per the per-topic §"Ethical-guard state" spec,
  and the Stage 3 session inherits the OFF default per §"Stage 3
  Synthesis Spec" → §"Discipline" → "Per topic-module ethical-guard
  inheritance".

**`timing-overlay` dispatch protocol**:

- `timing-overlay` is **not** a Stage 2 太极点 module; it overlays
  across the other in-scope modules and inherits their ethical-guard
  state (per §"Topic: timing-overlay" → §"How it composes with other
  modules").
- Dispatched **after** the other Stage 2 modules complete (so the
  overlay has its in-scope topic targets to compose against); produces
  annotation output rather than a standalone 太极点 topic file.
  Output path: `cases/case-NN-timing-overlay.md`.
- Routes through `rules/timing-techniques.md` + this file §"Topic:
  timing-overlay". If chosen in mode (d), append the overlay dispatch
  as step 5 to the dispatch sequence (after the Stage 2 parallel batch,
  before — or alongside — the Stage 3 synthesis session, depending on
  whether the overlay is to be folded into the portrait Frame).

### Parallel-dispatch boundaries

- **Stage 1 audit session is independent of the Stage 1 session.** It
  is a fresh-eye verification, dispatched as a different sub-session
  even if the same physical executor runs both (per
  `rules/stage1-data-extraction.md` §"Stage 1 → Stage 2 Handoff
  Protocol" → "Why a separate audit session").
- **Stage 2 sessions run in parallel within a mode.** Each Stage 2
  reads `chart-NN-stage1.md` as authoritative; there is no
  inter-Stage-2 dependency (per §4.1.5 "Module independence 原则" rule
  4). All Stage 2 modules in the chosen set can be dispatched as
  concurrent sessions once audit passes.
- **Stage 3 synthesis runs sequentially** — never concurrent with
  Stage 2 sessions still in flight (per §"Stage 3 Synthesis Spec" →
  §"Trigger" partial-input warning: synthesis with missing Stage 2
  modules introduces synthesis-from-incomplete-evidence bias).
- **Reader render is practitioner-triggered, not auto-dispatched** (SUPERSEDES
  原 "TL pass is parallel-safe within a mode" 段, 2026-05-18). Render
  跑哪个 topic / 何时跑 / 跑单 topic vs 全盘合并, 全部由 用户显式触发
  + render skill `Prerequisite` A/B 档判定. Render 不进入 Stage 2 ↔
  Stage 3 dispatch ordering 表 (它不是 pipeline 自动一环). 详
  `hellenistic-render-unguarded/SKILL.md`. — 原 TL pass parallel-safe
  陈述属于 v2.2 stage-based architecture 历史, 在本 fork 已 SUPERSEDED.

### Failure-mode dispatch rules

- **Stage 1 audit fail** → STOP the whole flow; fix Stage 1 per the
  audit corrections list; re-audit. Do **not** dispatch any Stage 2
  session until audit pass (per `rules/stage1-data-extraction.md`
  §"Stage 1 → Stage 2 Handoff Protocol" steps 3 / 5).
- **Stage 2 module fail** (output-discipline violation per §"Stage 2
  Session Output Format" → §"Discipline"; an in-advance Gate 3
  trace-back violation; ethical-guard inheritance breach; brief-shot
  8-criteria miss; etc.) → file an open thread on the offending Stage
  2 file + request a Stage 2 re-run; do **not** let Stage 3 consume
  the failed module's output. This is the no-silent-Stage-1-fix
  discipline (per §"Stage 2 Session Output Format" → §"Discipline")
  mirrored at the Stage 2 surface.
- **Stage 3 Gate 3 violation** → STOP the Stage 3 session; resolve the
  trace-back gap (either fix the Stage 3 evidence chain or fix the
  Stage 1 row per §"Stage 3 Synthesis Spec" → §"Discipline" "Gate 3
  enforcement"); re-run the Stage 3 session after the source is
  corrected. Gate 3 is a mechanical STOP-and-fix protocol, not a
  best-effort cite.
- **Sensitive-module consent breach** (a `death` / `longevity` Stage
  2 session dispatched without verified consent / without
  symbolic-frame declaration) → STOP the offending Stage 2 session;
  do **not** include its output in the Stage 3 portrait; mark the
  module as skipped-on-consent-failure in any portrait Frame that
  would have referenced it. Re-dispatch only after the consent +
  frame state is correctly recorded.
