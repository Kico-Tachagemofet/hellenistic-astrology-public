## 核心问题: euphemistic meta-category language

最常见的失败 mode 是 **抽象 meta-category 委婉**, 满足 "subject = native / person / event / domain" anti-cookbook 规则但**永远不落到具体 relational/event-level claim**.

### case-10 family 段失败案例

```
❌ 接受家是"结构性扛起来的承担"而不强求温情共生; 以承担一部分但
   不独自全扛的方式跟家保持节奏; 接受母性那层"贴在自我门口、跟内感
   与驱动捆在一起"的事实; 接受父性那层走"从对面 / 一段距离照过来"
   的间接通道。
```

subject grammar OK, 但读完读者**不知道家庭里在发生什么** — 谁吵架、谁疏远、什么冷、实际事件是什么. 用户反馈: "他都没有直接给我讲, 可能和父母关系不好, 然后会在家庭中比较容易生气, 然后, 和家庭有争吵, 或者是有破裂".

### 正确的 register

```
✓ 你跟父母关系是结构性紧张的 — 4 主火星塌到自我门口 + 火星整盘最
  难 (出境凶星 + 逆行). 在家容易点火、容易跟家人起冲突; 情绪推动
  和行动推动绑在一起的体质叠到家庭根基上, 最直接的展开就是
  **在家人面前点火快、跟家人争吵频率比对外人高**.
  父亲那条线疏远 (太阳 7 室 + 跟自轴跨轴对冲), 不是热的、近的、
  贴身指导型父子关系. 母亲那条线粘得太近 (月亮 1 室 + 跟火星紧合),
  亲密但也意味着情绪和动作模式跟母亲高度绞合, 容易冲突也容易粘黏.
  家庭里有关系破裂或长期张力的可能性偏高 — 这不是"避免不了",
  是"结构性偏高".
```

## Banned euphemism categories

life-claim 句子里禁用:

- ❌ **"结构性的责任依托" / "承担框架" / "承重的形态"** — 写**扛了什么 + 扛起来会怎样**: "他在家里要负责安排父母的医疗、自己挣的钱要补贴弟妹学费、同时还要在事业里独立"
- ❌ **"心理模式" / "行动模式"** — 写实际反复发生的场景: "他在压力下会先沉默几天、然后突然爆发、然后又自责"
- ❌ **"结构性张力" / "结构性疏远" / "结构性冲突" 单用** — 必须续具体场景: "结构性疏远 — 父亲长期不在生活里、关键时刻不会主动联系、案主也不会主动求父亲的意见"
- ❌ **"需要承担" / "需要面对" / "需要整合" 整句结论** — abstract task-language, 没说具体什么
- ❌ **"可能 / 也许 / 倾向于 / 容易出现" hedge 堆 + 后面没具体场景** — 证据够就 drop hedge 堆; 不够就降 confidence tier 明说
- ❌ **"走 X 路线 / 跑 Y 通道 / 在 Z 领域运作" 整句结论** — 这是 structural-language, 可以在 trailing citation 出现, 不能是 narrative 主体

(更全面的禁句式 / 排比对仗 / 抗重复 / 句子纪律见 `anti-cliche-astrology.md`)

## Required register: 5 类具体 claim

reader 散文里每个 life-claim 句子必须包含至少一种:

1. **concrete relational claim** ("和父母关系紧张" / "和兄弟姐妹竞争激烈" / "和合作伙伴反复破裂")
2. **concrete event-class claim** ("反复经历职业启动失败" / "在 X 年会面临婚姻冲突" / "身体在腰背处反复出问题")
3. **concrete behavioral claim** ("容易发火" / "会沉默回避" / "在公开场合压低自己")
4. **concrete material-life claim** ("生计走自营 / 顾问 / 慢工" / "财富积累偏慢但稳" / "会有一次大破财在 X 年")
5. **concrete identity/role claim** ("是家里的承担者" / "在伴侣眼里是被依赖的那个" / "在朋友圈是边缘观察者")

structural language (planet / sign / house / aspect / dignity / ruler chain) 放 **trailing citation block**, 不放句子主体.

## Body / illness / death / longevity 直接陈述模板

unguarded fork 把这 4 个 topic 当 ordinary topic 处理. 直接陈述, 不 hedge, 不 disclaimer:

- **Body**: "**身体结构性偏脆** 在 X 部位 — 因为 [evidence chain]. 实际可能表现: [具体的反复发生的身体事件]"
- **Illness**: "**会反复经历** [type of illness pattern] — 因为 [chain]. 风险窗口在 [profected / ZR 时段]"
- **Death cause**: "死亡方式从盘看偏向 [Hellenistic death-cause framework per Valens III / Firmicus VII] — [chain]. 这是**从盘的结构判断**, 不是预言时间."
- **Longevity**: "寿命承纳预估 [Hellenistic longevity technique: apheta-anareta + planetary years + ZR fortune] = **XX 到 XX 岁范围**. 主要支持证据: [chain]. 主要削减证据: [chain]."

confidence tier 按**实际证据强度** (≥2-source triangulation + audit 链), **不按 topic 类型**自动 downgrade. body/death/longevity finding 有 solid 审计基础时报 High 或 Moderate, 不被强制压成 Speculative.

---

## 无 disclaimer 段

reader 输出**不用** mandatory disclaimer 段:
- "伦理护栏复述"
- "本盘不作身体断言"
- "仅作符号 / 结构层级陈述"
- "绝不适合 父母身体 / 健康 / 死亡 / 寿命的预测断言"

finding 要么证据支持就写, 要么不写 / 降 confidence tier 明说. disclaimer 段做 evidence-tier 诚实度的替代品已被移除.

如果 practitioner 想给外部分享场景加 disclaimer, 那是 editorial 层事后加的事, 不是 finding 层 discipline.

## 时间节点括号写"为什么"

guarded 变体的 citation 习惯是 trailing finding-id 方括号: `[TF-character-01 / TF-chart-signature-06]`. 这对 technical 层 (Stage 2 finding / Stage 3 chart-finding-index) OK.

**但 reader 层** (reader/NN-<slug>.md / reader-entry portrait / Q&A 答), time-period claim 和重要 judgement 句必须含**括号 plain-language "为什么"**, 不只是 finding-id 指针:

```
❌ Reader 层只有 finding-id:
   "2028-2031 年是事业重定向窗口 [F-career-04 / F-timing-02]"
   → 读者不知道为什么, 必须翻 finding-index 才能验证

✅ Reader 层带 plain-language "为什么":
   "2028-2031 年是事业重定向窗口 (这几年主年走到 10 宫,
    profected lord 是水星, 水星又是 2 室主 + 11 室主,
    所以是'事业 + 财务 + 社交资源' 三条线同时被激活的窗口)"
   → 读者直接能看懂为什么, 也能跟自己生活对照判断准不准

✅ 双轨格式 (推荐 — reader 友好 + 技术可追溯):
   "2028-2031 年是事业重定向窗口 (profected to 10H + Mercury
    activated as 2-ruler + 11-ruler) [F-career-04 / F-timing-02]"
   → 括号给"为什么"的简洁人话, trailing brackets 给 finding-id 索引
```

### 各层 citation style

| Layer | Citation style |
|---|---|
| Stage 2 topic-finding statements | Trailing brackets finding-id + Stage 1 row_id chain (technical) |
| Stage 3 chart-finding-index entries | Trailing brackets finding-id + cross-finding refs (technical) |
| **Reader topic .md (本 skill 输出层)** | Parenthesized plain-language why (primary) + 整文件底部 trailing block 完整 technical (audit trail) |
| Q&A mode answers | Parenthesized plain-language why (mandatory) + finding-id reference 翻译成人话 |

### Rule: 每个 reader 层时间节点

"这段时间会 X" / "在 YYYY 年" / "整个 ZR major period" 类句子必须含括号"为什么":

- ✅ "2028-2031 重承担窗口 (Saturn 8H 3 层 Jupiter-hosted + ZR Lot of Fortune 进 Saturn major period)"
- ✅ "30 岁那年伴侣关系有事 (主年 30 → profect to 7H + 7-ruler 状态 X)"
- ❌ "2028-2031 重承担窗口" — 干句, 读者不知道为什么
- ❌ "2028-2031 重承担窗口 [F-timing-04]" — finding-id 不是人话

### Rule: 每个 reader 层重要 judgement 句

同样标准:

- ✅ "你跟父母关系结构性紧张 (4 主火星塌缩到自我门口 + 火星是整盘最难的那颗 → 在家人面前点火快、争吵频率高)"
- ❌ "你跟父母关系结构性紧张" — 干结论
- ❌ "你跟父母关系结构性紧张 [TF-family-04]" — finding-id 不是人话
